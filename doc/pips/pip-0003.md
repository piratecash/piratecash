PIP-0003: Standard Transaction Size Re-Alignment
================================================

| Field | Value |
| ----- | ----- |
| PIP | 0003 |
| Title | Standard Transaction Size Re-Alignment |
| Status | Proposed |
| Type | Standards Track |
| Layer | Consensus (legacy ceiling) / Policy (relay, mempool, wallet) |
| Created | 2026-05-04 |
| Stage 1 target | v19 |
| Stage 2 target | v20 |
| Stage 3 target | v21 (height-gated, hardfork) |
| Affected files (v19) | `src/policy/policy.h`, `src/policy/policy.cpp`, `src/validation.cpp`, `src/net_processing.cpp`, `src/wallet/wallet.cpp`, `src/node/transaction.cpp`, `test/functional/mempool_accept.py` |
| Affected files (v20) | `src/policy/policy.cpp`, `src/net_processing.cpp`, `src/node/transaction.cpp` (guard removed), `test/functional/mempool_accept.py` |
| Affected files (v21) | `src/consensus/params.h`, `src/chainparams.cpp`, `src/validation.cpp` |

> This proposal re-aligns the maximum standard transaction size of
> PirateCash with upstream Dash/Cosanta in three steps. v19 (stage 1)
> stops the bleeding by introducing a soft policy that prevents new
> oversized transactions from originating on upgraded nodes while
> preserving full compatibility with pre-v19 peers and with the
> existing on-chain history. v20 (stage 2) tightens the relay/mempool
> path to the same limit. Stages 1 and 2 are deliberately **soft
> forks at most** — a v19/v20 node never rejects a block that an
> older node would have accepted. v21 (stage 3) introduces a
> height-gated consensus rule and is therefore an explicit
> **hardfork**: after a chain-params-defined activation height,
> blocks containing oversized transactions are rejected. Below the
> activation height consensus continues to apply the legacy ceiling,
> so initial sync of the historical chain remains possible.

## English

### Background

Upstream Dash (and Cosanta v19) caps standard transaction size at
`MAX_STANDARD_TX_SIZE = 100000` bytes. Historically, PirateCash raised
this constant to `2900000` bytes in `src/policy/policy.h`, and the
same constant is reused in five places:

- `src/policy/policy.h` — definition;
- `src/validation.cpp` (`ContextualCheckTransaction`) — **consensus**
  rule, applied to every block transaction during
  `ContextualCheckBlock` and to every mempool admission via
  `AcceptToMemoryPoolWorker`;
- `src/policy/policy.cpp` (`IsStandardTx`) — relay/mempool policy;
- `src/net_processing.cpp` — orphan-pool admission cap;
- `src/wallet/wallet.cpp` — refusal to construct an oversized
  transaction.

Because the same constant is consensus-critical, blindly lowering it
back to `100000` would cause every fully synchronised node to reject
the historical PirateCash chain and produce a network split.

### Goal

1. Match the Dash relay/mempool/wallet limit (`100000`) for new
   transactions while keeping the historical chain valid.
2. Avoid a hardfork in v19 and v20: the consensus rule stays
   permissive at the legacy ceiling for both releases. A hardfork is
   accepted as the right tool for stage 3 (v21), once the network
   has had time to upgrade.
3. Roll out gradually so older peers continue to interoperate during
   the transition.

### Two constants

`src/policy/policy.h` defines:

```cpp
/** Strict, Dash-aligned. Used by the wallet (today) and by
 *  mempool/relay/orphan paths starting with v20. */
static const unsigned int MAX_STANDARD_TX_SIZE = 100000;

/** Legacy ceiling, kept forever for consensus, and temporarily
 *  reused by mempool/relay/orphan paths in v19. */
static const unsigned int MAX_LEGACY_TX_SIZE = 2900000;
```

The exact value `2900000` matches the historical PirateCash constant
and gives sufficient headroom over the largest oversized transaction
already on chain.

### Stage 1 — v19 (this release)

Variant **B** ("soft mempool") is implemented in this release.

| Use site | File / line | Constant used |
| --- | --- | --- |
| Block & mempool consensus check | `src/validation.cpp` (`ContextualCheckTransaction`) | `MAX_LEGACY_TX_SIZE` |
| Relay / mempool policy (`IsStandardTx`) | `src/policy/policy.cpp` | `MAX_LEGACY_TX_SIZE` |
| Orphan-pool admission | `src/net_processing.cpp` | `MAX_LEGACY_TX_SIZE` |
| **Local submission guard** (`BroadcastTransaction`) | `src/node/transaction.cpp` | **`MAX_STANDARD_TX_SIZE`** |
| Wallet creation | `src/wallet/wallet.cpp` | **`MAX_STANDARD_TX_SIZE`** |

The `BroadcastTransaction` guard matters because `IsStandardTx` is
soft in v19. Without it, a hand-crafted oversized transaction
submitted via `sendrawtransaction` (or any other local RPC entry
point that funnels through `BroadcastTransaction`) would still slip
through the mempool. Locking down `BroadcastTransaction` means
**no new oversized transaction originates from a v19 node**, even
when the wallet is bypassed — which is the actual guarantee this
PIP claims for stage 1. P2P-relayed legacy-sized transactions from
older peers continue to traverse the v19 mempool unaffected.

Effect:

- Initial sync of the historical chain is unaffected (consensus is
  satisfied by the legacy ceiling).
- Pre-v19 peers can still relay legacy-sized transactions through a
  v19 node — `IsStandardTx` and the orphan path are gated on
  `MAX_LEGACY_TX_SIZE`.
- An upgraded node will refuse both to *create* (wallet) and to
  *locally submit* (sendrawtransaction) an oversized transaction —
  the source of new oversized transactions dries up.

### Stage 2 — v20 (next release)

The mempool/relay/orphan use sites switch to the strict limit, and
the now-redundant `BroadcastTransaction` guard is removed:

| Use site | v19 | v20 |
| --- | --- | --- |
| `ContextualCheckTransaction` (consensus) | `MAX_LEGACY_TX_SIZE` | `MAX_LEGACY_TX_SIZE` (unchanged) |
| `IsStandardTx` (policy) | `MAX_LEGACY_TX_SIZE` | `MAX_STANDARD_TX_SIZE` |
| Orphan-pool admission | `MAX_LEGACY_TX_SIZE` | `MAX_STANDARD_TX_SIZE` |
| `BroadcastTransaction` local guard | `MAX_STANDARD_TX_SIZE` | **removed** (redundant once `IsStandardTx` is strict) |
| Wallet creation | `MAX_STANDARD_TX_SIZE` | `MAX_STANDARD_TX_SIZE` (unchanged) |

The change in v20 is intentionally trivial: replace `MAX_LEGACY_TX_SIZE`
with `MAX_STANDARD_TX_SIZE` in `src/policy/policy.cpp` and
`src/net_processing.cpp`, drop the guard block in
`src/node/transaction.cpp`, restore the `64: tx-size` expectation in
`test/functional/mempool_accept.py`, and update the surrounding
comments. `MAX_LEGACY_TX_SIZE` itself stays defined because consensus
still references it.

### Stage 3 — v21 (height-gated consensus tightening, hardfork)

After v20 has had time to roll out across the network, v21 introduces
a height-gated consensus rule. After a chain-params-defined activation
height — call it `nTxSizeStandardEnforcementHeight` — blocks
containing any transaction larger than `MAX_STANDARD_TX_SIZE` are
rejected at consensus level. Below the activation height the legacy
ceiling continues to apply, so initial sync of the historical chain is
still possible.

**The exact activation height is intentionally not specified in this
PIP.** It must be picked closer to the v21 release date, with enough
headroom over the current chain tip so that all live nodes have time
to upgrade. The height will be added to `Consensus::Params` for each
network (mainnet, testnet, devnet, regtest) in `src/chainparams.cpp`
and recorded in this PIP at that time.

Sketch of the consensus change in `ContextualCheckTransaction`
(`src/validation.cpp`):

```cpp
const unsigned int nMaxTxSize =
    (consensusParams.nTxSizeStandardEnforcementHeight != 0 &&
     nHeight >= consensusParams.nTxSizeStandardEnforcementHeight)
    ? MAX_STANDARD_TX_SIZE
    : MAX_LEGACY_TX_SIZE;

if (fDIP0001Active_context && ::GetSerializeSize(tx, PROTOCOL_VERSION) > nMaxTxSize)
    return state.Invalid(ValidationInvalidReason::CONSENSUS, false,
                         REJECT_INVALID, "bad-txns-oversize");
```

Setting the activation height to `0` keeps the legacy ceiling and is
the recommended value for `regtest` so existing test fixtures keep
working unchanged.

#### Compatibility and hardfork classification

This is a hardfork: a v20 (or earlier) node will accept a block that a
v21 node rejects, once the activation height is reached. The rollout
plan is therefore the same as for any other consensus bump in
PirateCash:

- announce the activation height in release notes well before it is
  reached;
- coordinate with masternode operators and exchanges through the same
  channels used for previous protocol bumps;
- bump the protocol version so v21 nodes prefer v21 peers around the
  switch.

By the time v21 ships, no honestly-built transaction should be
oversized anyway — wallets stopped producing them in v19, and v20
relay/mempool refuse to propagate them — so the rule is expected to
be a no-op on the honest chain.

### Alternatives considered

- **Variant A — strict mempool already in v19.** Switch
  `IsStandardTx` and the orphan path to `MAX_STANDARD_TX_SIZE`
  immediately. Rejected for v19 because pre-v19 peers can still
  rebroadcast legacy-sized transactions and we did not want the v19
  upgrade to look like a P2P regression. Variant A is exactly what
  Stage 2 (v20) does once the network has had time to upgrade.
- **Hardfork to lower the consensus ceiling.** Rejected as
  unnecessary for the goal stated in this PIP and disproportionately
  risky for v19.

### Testing

- Existing test `src/wallet/test/coinselector_tests.cpp` constructs
  676 inputs ≈ 100k bytes, which already targets the strict limit;
  no change required.
- `src/test/transaction_tests.cpp` does not depend on the absolute
  value of `MAX_STANDARD_TX_SIZE`.
- `test/functional/mempool_accept.py` had a hard-coded `100000` in
  the `tx-size` reject sub-case. In v19 this sub-case necessarily
  shifts to the consensus reject path — `ContextualCheckTransaction`
  runs before `IsStandardTx` in `AcceptToMemoryPoolWorker` and
  rejects a tx larger than `MAX_LEGACY_TX_SIZE` with
  `16: bad-txns-oversize` before the `64: tx-size` policy path is
  reached. The test was therefore updated to build a tx above
  `MAX_LEGACY_TX_SIZE` and to expect the consensus reject. In v20,
  with `IsStandardTx` back on `MAX_STANDARD_TX_SIZE`, the test
  returns to the `64: tx-size` expectation with a tx slightly above
  `MAX_STANDARD_TX_SIZE`.
- Other functional tests under `test/functional/` reuse
  `MAX_BLOCK_SIZE`, not `MAX_STANDARD_TX_SIZE`; no change required.
- `test/functional/mempool_accept.py` was extended with a sub-case
  that submits a transaction sized `MAX_STANDARD_TX_SIZE + 1` (still
  well below `MAX_LEGACY_TX_SIZE`) via `sendrawtransaction` and
  asserts the v19 `BroadcastTransaction` guard rejects it with
  RPC error code `-26` and message `tx-size-local`. This sub-case
  is removed in v20 along with the guard itself.
- Regression check before tagging v19: run mainnet headers-only sync
  and a partial block sync against a node holding the historical
  chain; confirm `bad-txns-oversize` is not raised.
- Regression check before tagging v20: same as above, plus verify
  that newly built v20 relay rejects an injected legacy-sized tx and
  that the orphan path drops it.

## Русский

### Контекст

Upstream Dash (и Cosanta v19) ограничивают размер стандартной
транзакции константой `MAX_STANDARD_TX_SIZE = 100000` байт.
Исторически в PirateCash эта константа в `src/policy/policy.h` была
поднята до `2900000` байт, и она же используется в пяти местах:

- `src/policy/policy.h` — определение;
- `src/validation.cpp` (`ContextualCheckTransaction`) — **consensus**,
  применяется к каждой транзакции в блоке через `ContextualCheckBlock`
  и к каждому добавлению в мемпул через `AcceptToMemoryPoolWorker`;
- `src/policy/policy.cpp` (`IsStandardTx`) — policy для relay/mempool;
- `src/net_processing.cpp` — лимит на размер orphan-tx;
- `src/wallet/wallet.cpp` — запрет на создание жирной транзакции.

Поскольку та же самая константа критична для consensus, простое
понижение её обратно до `100000` приведёт к тому, что любая
полностью синхронизированная нода отвергнет историческую цепь
PirateCash и сеть расколется.

### Цель

1. Привести лимит relay/mempool/wallet к значению Dash (`100000`) для
   новых транзакций, не ломая историческую цепь.
2. Никакого хардфорка в v19 и v20: consensus-правило в обоих
   релизах остаётся либеральным на уровне legacy ceiling. Хардфорк
   принимается как уместный инструмент для этапа 3 (v21), когда
   сеть успеет обновиться.
3. Постепенный rollout, чтобы старые пиры продолжали взаимодействовать
   во время перехода.

### Две константы

`src/policy/policy.h` определяет:

```cpp
/** Строгий лимит, согласован с Dash. Сейчас используется только
 *  кошельком; начиная с v20 — также мемпулом / relay / orphan-пулом. */
static const unsigned int MAX_STANDARD_TX_SIZE = 100000;

/** Legacy потолок. Остаётся навсегда для consensus, в v19 временно
 *  переиспользуется в мемпуле / relay / orphan-пуле. */
static const unsigned int MAX_LEGACY_TX_SIZE = 2900000;
```

Значение `2900000` совпадает с исторической константой PirateCash и
даёт запас над самой жирной транзакцией, уже находящейся в цепи.

### Этап 1 — v19 (текущий релиз)

В этом релизе реализован вариант **B** («мягкий мемпул»).

| Точка использования | Файл | Константа |
| --- | --- | --- |
| Consensus-проверка блока и мемпула | `src/validation.cpp` (`ContextualCheckTransaction`) | `MAX_LEGACY_TX_SIZE` |
| Policy relay / mempool (`IsStandardTx`) | `src/policy/policy.cpp` | `MAX_LEGACY_TX_SIZE` |
| Orphan-пул | `src/net_processing.cpp` | `MAX_LEGACY_TX_SIZE` |
| **Локальный submission-guard** (`BroadcastTransaction`) | `src/node/transaction.cpp` | **`MAX_STANDARD_TX_SIZE`** |
| Создание транзакции в кошельке | `src/wallet/wallet.cpp` | **`MAX_STANDARD_TX_SIZE`** |

Guard в `BroadcastTransaction` важен именно потому, что
`IsStandardTx` в v19 мягкий. Без него ручную жирную транзакцию,
поданную через `sendrawtransaction` (или любой другой локальный RPC,
который заводит транзакцию через `BroadcastTransaction`), мемпул
пропустил бы. С guard'ом **новая жирная транзакция не появляется
из v19-ноды ни одним путём** — даже в обход кошелька, — что и есть
реальная гарантия, которую этот PIP заявляет для этапа 1. P2P-релэй
legacy-tx от старых пиров через v19-мемпул при этом не страдает.

Эффект:

- Начальная синхронизация исторической цепи не страдает (consensus
  удовлетворяется legacy-потолком).
- Пре-v19 пиры по-прежнему могут релэить legacy-tx через v19-ноду —
  `IsStandardTx` и orphan-путь v19 завязаны на `MAX_LEGACY_TX_SIZE`.
- Обновлённая нода отказывается и *создавать* жирную транзакцию
  (кошелёк), и *локально подавать* её (sendrawtransaction) —
  источник новых жирных транзакций иссякает.

### Этап 2 — v20 (следующий релиз)

Точки использования mempool/relay/orphan переключаются на строгий
лимит, и одновременно удаляется ставший избыточным guard в
`BroadcastTransaction`:

| Точка использования | v19 | v20 |
| --- | --- | --- |
| `ContextualCheckTransaction` (consensus) | `MAX_LEGACY_TX_SIZE` | `MAX_LEGACY_TX_SIZE` (без изменений) |
| `IsStandardTx` (policy) | `MAX_LEGACY_TX_SIZE` | `MAX_STANDARD_TX_SIZE` |
| Orphan-пул | `MAX_LEGACY_TX_SIZE` | `MAX_STANDARD_TX_SIZE` |
| Локальный guard в `BroadcastTransaction` | `MAX_STANDARD_TX_SIZE` | **удалён** (избыточен после правки `IsStandardTx`) |
| Кошелёк | `MAX_STANDARD_TX_SIZE` | `MAX_STANDARD_TX_SIZE` (без изменений) |

Изменение в v20 намеренно тривиально: заменить `MAX_LEGACY_TX_SIZE`
на `MAX_STANDARD_TX_SIZE` в `src/policy/policy.cpp` и
`src/net_processing.cpp`, выпилить guard-блок в
`src/node/transaction.cpp`, вернуть ожидание `64: tx-size` в
`test/functional/mempool_accept.py`, обновить комментарии. Сама
`MAX_LEGACY_TX_SIZE` остаётся определённой — на неё опирается
consensus.

### Этап 3 — v21 (height-gated consensus tightening, хардфорк)

После того как v20 успеет распространиться по сети, v21 вводит
height-gated consensus-правило. Начиная с активационной высоты,
заданной в chain params — назовём её
`nTxSizeStandardEnforcementHeight`, — блоки, содержащие хотя бы
одну транзакцию больше `MAX_STANDARD_TX_SIZE`, отвергаются на уровне
consensus. До активационной высоты продолжает применяться legacy
ceiling, чтобы синхронизация исторической цепи оставалась возможной.

**Конкретная активационная высота в этом PIP сознательно не
указывается.** Её надо подобрать ближе к релизу v21 с достаточным
запасом над текущим тиром цепи, чтобы все живые ноды успели
обновиться. Высота добавляется в `Consensus::Params` для всех сетей
(mainnet, testnet, devnet, regtest) в `src/chainparams.cpp` и
фиксируется в этом PIP в момент релиза.

Скетч consensus-изменения в `ContextualCheckTransaction`
(`src/validation.cpp`):

```cpp
const unsigned int nMaxTxSize =
    (consensusParams.nTxSizeStandardEnforcementHeight != 0 &&
     nHeight >= consensusParams.nTxSizeStandardEnforcementHeight)
    ? MAX_STANDARD_TX_SIZE
    : MAX_LEGACY_TX_SIZE;

if (fDIP0001Active_context && ::GetSerializeSize(tx, PROTOCOL_VERSION) > nMaxTxSize)
    return state.Invalid(ValidationInvalidReason::CONSENSUS, false,
                         REJECT_INVALID, "bad-txns-oversize");
```

Активационную высоту `0` следует трактовать как «правило не
включено» и использовать в `regtest`, чтобы существующие фикстуры
тестов продолжали работать без правок.

#### Совместимость и классификация хардфорка

Это хардфорк: v20 (и более ранние) ноды примут блок, который v21
отвергнет, когда активационная высота будет достигнута. План rollout
тот же, что и для других consensus-бампов в PirateCash:

- объявить активационную высоту в release notes заранее, задолго до
  её достижения;
- скоординировать с операторами мастернод и биржами по тем же
  каналам, что и предыдущие протокол-бампы;
- поднять protocol version, чтобы v21-ноды у момента переключения
  предпочитали v21-пиров.

К моменту v21 в сети уже не должно быть честно построенных жирных
транзакций — кошельки перестали их создавать в v19, а relay/mempool
не пропагируют их с v20 — поэтому правило ожидаемо будет no-op на
честной цепи.

### Рассмотренные альтернативы

- **Вариант A — строгий мемпул уже в v19.** Сразу переключить
  `IsStandardTx` и orphan-путь на `MAX_STANDARD_TX_SIZE`. Отклонено
  для v19, потому что пре-v19 пиры всё ещё могут ребродкастить
  legacy-tx, и v19-апгрейд не должен выглядеть как P2P-регрессия.
  Вариант A — это в точности то, что делает этап 2 (v20), когда сеть
  успеет обновиться.
- **Хардфорк с понижением consensus-потолка.** Отклонено как
  избыточное для задачи и непропорционально рисковое для v19.

### Тестирование

- Существующий тест `src/wallet/test/coinselector_tests.cpp`
  строит 676 входов ≈ 100k байт — он уже целится в строгий лимит,
  изменения не нужны.
- `src/test/transaction_tests.cpp` не зависит от абсолютного значения
  `MAX_STANDARD_TX_SIZE`.
- В `test/functional/mempool_accept.py` была захардкожена цифра
  `100000` в саб-кейсе с reject `tx-size`. В v19 этот саб-кейс
  неизбежно смещается на consensus reject: в
  `AcceptToMemoryPoolWorker` `ContextualCheckTransaction` выполняется
  до `IsStandardTx` и отвергает транзакцию размером больше
  `MAX_LEGACY_TX_SIZE` с reason `16: bad-txns-oversize` ещё до того,
  как сработает policy-путь `64: tx-size`. Тест поэтому обновлён так,
  что строит транзакцию выше `MAX_LEGACY_TX_SIZE` и ожидает именно
  consensus reject. В v20, когда `IsStandardTx` вернётся на
  `MAX_STANDARD_TX_SIZE`, тест возвращается к ожиданию
  `64: tx-size` с транзакцией чуть выше `MAX_STANDARD_TX_SIZE`.
- Прочие функциональные тесты `test/functional/` используют
  `MAX_BLOCK_SIZE`, а не `MAX_STANDARD_TX_SIZE`; изменения не нужны.
- В `test/functional/mempool_accept.py` добавлен саб-кейс, где
  транзакция размером `MAX_STANDARD_TX_SIZE + 1` (всё ещё далеко
  ниже `MAX_LEGACY_TX_SIZE`) подаётся через `sendrawtransaction` и
  ожидается reject новым guard'ом `BroadcastTransaction` с
  RPC-кодом ошибки `-26` и сообщением `tx-size-local`. В v20 этот
  саб-кейс удаляется одновременно с самим guard'ом.
- Регресс-проверка перед тегом v19: headers-only-синхронизация в
  mainnet и частичная синхронизация блоков против ноды с исторической
  цепью; убедиться, что не возникает `bad-txns-oversize`.
- Регресс-проверка перед тегом v20: то же самое, плюс инжект
  legacy-tx в свежесобранный v20-relay и проверка, что мемпул её
  отвергает, а orphan-путь её дропает.
