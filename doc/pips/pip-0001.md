## English

### Implementation status

This block mirrors the BIP / Dash DIP convention of tracking each
deployment stage separately rather than flipping a single proposal-wide
flag.

| Stage | Description | Status | Target release |
| ----- | ----------- | ------ | -------------- |
| 1 | Operator-configuration gate, startup probe of `/rpc/v1/system/node_status`, hourly heartbeat monitor on active masternodes | **Active** | PirateCash Core v19 |
| 2 | Network-visible Corsa availability checks (`ping`, `version`, peer health) | Draft | PirateCash Core v20 |
| 3 | PoSe enforcement using a signed proof of Corsa service tied to the masternode operator key | Draft | PirateCash Core v21 |

Once Stage 2 and Stage 3 land, this table is updated in place — the
proposal-level `Status` field is bumped to `Final` only after the last
stage is activated on mainnet.

### Abstract

PirateCash announced a decentralized messenger. The messenger is implemented in
a separate repository as Corsa because it must be able to evolve without forcing
constant PirateCash Core network upgrades. This proposal defines a staged plan
for making Corsa service availability part of PirateCash masternode operation.

### Motivation

The messenger needs a stable set of always-on network operators. PirateCash
masternodes already provide incentivized infrastructure, so they are the natural
place to host the messenger service.

Keeping Corsa outside the Core repository lets the messenger protocol, desktop
application, node implementation, and API improve independently. PirateCash Core
should only enforce the minimum integration points that are required for network
reliability.

### Definitions

- `PirateCash Core node`: the PirateCash daemon or GUI process that manages the
  masternode.
- `Corsa node`: the standalone messenger node process from the Corsa repository.
- `Corsa API`: the local HTTP RPC API exposed by a Corsa node. The current Corsa
  documentation describes `POST /rpc/v1/exec`, including commands such as
  `ping`, `version`, `getPeers`, `fetchPeerHealth`, and `fetchNetworkStats`.
- `Service proof`: evidence that a masternode is running an acceptable Corsa
  service. The exact proof format is intentionally left for the v21
  implementation design.

### Specification

The integration is deployed in three stages.

#### Stage 1: v19 local requirement

Starting with PirateCash Core v19, a masternode must not start unless the
operator explicitly configures the RPC username, RPC password, and RPC port used
for the local Corsa service check.

The Corsa node must be configured on the same server as the PirateCash
masternode. The operator only configures the local Corsa RPC port —
PirateCash Core always connects to `127.0.0.1`, so HTTP Basic
credentials never traverse the network. At startup, PirateCash Core
verifies that the messenger node is present, reachable, and that its
RPC server enforces authentication: a second probe with deliberately
invalid credentials must be rejected with HTTP `401`/`403`. A Corsa
node that accepts the bogus request (auth disabled) fails the gate and
the masternode does not start. This stage does not verify public
service quality, does not verify relay performance, and does not apply
PoSe penalties.

The v19 check should be treated as an operator configuration gate. Its purpose is
to make sure masternode operators install and expose the local Corsa service
before stronger network-level checks are introduced.

#### Stage 2: v20 API availability checks

Starting with PirateCash Core v20, nodes should verify Corsa service
availability through the Corsa API.

The check should use authenticated local RPC access and should validate at least
basic node liveness and version information. Suitable API commands include
`ping` and `version`; service-health commands such as peer health or network
statistics may be used when the implementation is ready.

This stage should make Corsa availability observable to the network, but it
should still be conservative about penalties until the proof mechanism is stable.

#### Stage 3: v21 PoSe enforcement

Starting with PirateCash Core v21, masternodes should receive PoSe score when
they fail to provide an acceptable proof of Corsa service.

The proof must be resistant to trivial spoofing, must be tied to the masternode
identity or operator-controlled key material, and must be verifiable by other
nodes without trusting the operator's local machine.

### Compatibility

v19 is an operational requirement for masternode startup and should not change
block or transaction consensus rules.

v20 introduces network-visible service checks and should remain compatible with
the v19 configuration model.

v21 changes masternode punishment behavior through PoSe and therefore needs
clear activation rules, release notes, and operator migration time.

### Security considerations

RPC credentials must not be printed to logs, debug output, RPC errors, or GUI
messages.

The Corsa API should be bound to localhost by default and should require
authentication when PirateCash Core uses it for masternode checks.

Remote checks must be rate-limited and designed to avoid turning masternodes
into amplification targets.

The v21 service proof must not be satisfied by a simple open port. It should
prove that the operator controls a working Corsa node that can participate in the
messenger network.

### Rationale

The staged rollout avoids forcing immediate punishment for operators while the
messenger service is still being integrated. v19 makes the requirement visible,
v20 makes the service observable, and v21 makes the service enforceable through
PoSe.

## Русский

### Статус реализации

Блок повторяет соглашение BIP / Dash DIP: статус трекается по каждому
этапу отдельно, а не одним общим флагом для всего предложения.

| Этап | Описание | Статус | Целевой релиз |
| ---- | -------- | ------ | ------------- |
| 1 | Операторская конфигурация-гейт, стартовый probe `/rpc/v1/system/node_status`, ежечасный heartbeat-монитор на активных мастернодах | **Active** | PirateCash Core v19 |
| 2 | Сетевые проверки доступности Corsa (`ping`, `version`, peer health) | Draft | PirateCash Core v20 |
| 3 | Применение PoSe по подписанному proof-of-service, привязанному к operator-ключу мастерноды | Draft | PirateCash Core v21 |

После выкатки этапов 2 и 3 обновляется именно эта таблица; общий
`Status` предложения переводится в `Final` только когда последний этап
активирован в mainnet.

### Аннотация

PirateCash анонсировал децентрализованный мессенджер. Мессенджер реализован в
отдельном репозитории как Corsa, потому что он должен развиваться без
постоянных обязательных обновлений всей сети PirateCash Core. Это предложение
описывает поэтапный план, как сделать доступность сервиса Corsa частью работы
мастернод PirateCash.

### Мотивация

Мессенджеру нужна стабильная инфраструктура постоянно работающих узлов.
Мастерноды PirateCash уже являются мотивированными операторами сети, поэтому
они естественно подходят для размещения сервиса мессенджера.

Отдельный репозиторий Corsa позволяет обновлять протокол мессенджера, desktop
приложение, реализацию ноды и API независимо от PirateCash Core. PirateCash Core
должен фиксировать только минимальные точки интеграции, которые нужны для
надежности сети.

### Определения

- `PirateCash Core node`: daemon или GUI-процесс PirateCash, который управляет
  мастернодой.
- `Corsa node`: отдельный процесс ноды мессенджера из репозитория Corsa.
- `Corsa API`: локальный HTTP RPC API ноды Corsa. Текущая документация Corsa
  описывает `POST /rpc/v1/exec`, включая команды `ping`, `version`, `getPeers`,
  `fetchPeerHealth` и `fetchNetworkStats`.
- `Proof of service`: доказательство того, что мастернода запускает приемлемый
  сервис Corsa. Точный формат доказательства намеренно оставлен для дизайна
  реализации v21.

### Спецификация

Интеграция внедряется в три этапа.

#### Этап 1: локальное требование v19

Начиная с PirateCash Core v19, мастернода не должна запускаться, если оператор
явно не указал RPC-пользователя, RPC-пароль и RPC-порт для локальной проверки
сервиса Corsa.

Нода Corsa должна быть настроена на том же сервере, что и мастернода
PirateCash. Оператор задаёт только порт локального Corsa RPC; PirateCash
Core всегда подключается к `127.0.0.1`, поэтому HTTP Basic-credentials
не уходят в сеть. При запуске PirateCash Core проверяет, что нода
мессенджера существует, доступна и что её RPC-сервер действительно
требует аутентификацию: вторая проверка с заведомо неверными
credentials должна быть отклонена с HTTP `401`/`403`. Если Corsa
принимает запрос с фальшивыми credentials (аутентификация выключена) —
гейт не пройден и мастернода не запускается. На этом этапе не
проверяется публичное качество сервиса, не проверяется качество
ретрансляции и не применяется PoSe.

Проверку v19 следует считать операторским требованием конфигурации. Ее цель -
убедиться, что операторы мастернод установили и открыли локальный сервис Corsa
до внедрения более строгих сетевых проверок.

#### Этап 2: проверки доступности через API в v20

Начиная с PirateCash Core v20, ноды должны проверять доступность сервиса Corsa
через Corsa API.

Проверка должна использовать аутентифицированный локальный RPC-доступ и должна
подтверждать как минимум базовую доступность ноды и информацию о версии. Для
этого подходят команды API `ping` и `version`; команды состояния сервиса,
например health пиров или статистика сети, могут использоваться, когда
реализация будет готова.

На этом этапе доступность Corsa должна стать наблюдаемой для сети, но штрафы
следует вводить осторожно, пока механизм proof of service не стабилизирован.

#### Этап 3: применение PoSe в v21

Начиная с PirateCash Core v21, мастерноды должны получать PoSe score, если они
не предоставляют приемлемый proof of service для Corsa.

Доказательство должно быть устойчиво к простой подделке, должно быть связано с
identity мастерноды или ключевым материалом оператора и должно проверяться
другими нодами без доверия к локальной машине оператора.

### Совместимость

v19 является операционным требованием для запуска мастерноды и не должен менять
consensus-правила блоков или транзакций.

v20 добавляет видимые для сети проверки сервиса и должен сохранять совместимость
с конфигурационной моделью v19.

v21 меняет поведение наказаний мастернод через PoSe, поэтому требует понятных
правил активации, release notes и времени для миграции операторов.

### Соображения безопасности

RPC-учетные данные нельзя выводить в логи, debug output, RPC-ошибки или GUI
сообщения.

Corsa API по умолчанию должен быть привязан к localhost и должен требовать
аутентификацию, когда PirateCash Core использует его для проверок мастернод.

Удаленные проверки должны иметь rate limit и не должны превращать мастерноды в
цели для amplification attacks.

Proof of service в v21 не должен сводиться к простому открытому порту. Он должен
доказывать, что оператор контролирует рабочую ноду Corsa, которая может
участвовать в сети мессенджера.

### Обоснование

Поэтапное внедрение не заставляет сразу наказывать операторов, пока сервис
мессенджера еще интегрируется. v19 делает требование явным, v20 делает сервис
наблюдаемым, а v21 делает сервис обязательным через PoSe.
